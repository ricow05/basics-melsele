The site needs to have a login system, I also need a database, authorized people also need to be able to add simple things to the website and database from the frontend.

Then GitHub Pages is the wrong fit.

Your requirements mean you need:
1. Authentication/login
2. A real database
3. Role-based access for authorized users
4. A frontend where authorized users can add or edit content
5. Secure server-side logic for writes and permissions

A good setup for this is:

1. Vercel or Netlify for the website
- Hosts your React/Vite frontend
- Easy Git-based deploys
- Supports backend functions if needed

2. Supabase for backend
- PostgreSQL database
- Built-in auth
- Role-based access with row-level security
- File storage if you later need images/docs
- Simple JavaScript client for frontend integration

Why this is probably the best fit for you:
- Fastest path from your current Vite site
- You get login + database without building a backend from scratch
- Authorized users can manage content from your site’s frontend
- Much simpler than running your own server

What the architecture would look like:
- Frontend: your existing React/Vite site
- Hosting: Vercel
- Auth: Supabase Auth
- Database: Supabase Postgres
- Admin features: protected pages in your frontend
- Permissions: Supabase policies so only approved users can create/edit/delete

Typical flow:
- Visitor opens site
- User logs in
- App checks whether user is admin/editor
- If authorized, they see a dashboard
- From that dashboard they can add news, agenda items, activities, sponsors, etc.
- Data is saved to Supabase and shown on the public site

What I would not recommend for this project:
- GitHub Pages alone: no backend, no secure auth, no database
- Firebase if you want simple relational data and SQL-like structure
- A custom VPS backend unless you want more maintenance

Best practical choice:
- Vercel + Supabase

Alternative if you want everything inside one ecosystem:
- Firebase Hosting + Firebase Auth + Firestore

Difference:
- Supabase is better if your data is structured like tables: users, events, news, sponsors, registrations
- Firebase is better if you want rapid app building and do not mind a NoSQL model

For your site, I would model it roughly like this:
- users
- roles
- news
- agenda_items
- activities
- sponsors
- pages_content

And these frontend areas:
- public pages
- login page
- admin dashboard
- forms to create/edit/delete items
